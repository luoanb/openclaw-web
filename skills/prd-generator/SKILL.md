# PRD / 产品功能文档生成

## 目标

- 与用户**多轮次**对齐：先收敛问题与范围，再落文档，避免一次性臆测。
- 产出**单一事实来源**：结构化 Markdown，便于评审、排期与拆 Story。
- 明确**验收标准**与**开放问题**，减少返工。

## 工作原则

1. **先复述再提问**：每轮开头用 3～6 句话复述当前已确认的事实；新增内容单独列出「待您确认」。
2. **分批提问**：每轮最多 **5～8 个**具体问题；复杂主题拆多轮，避免问卷墙。
3. **缺省不编造**：未知项写入文档「开放问题 / 假设」，并标注负责人或决策截止条件。
4. **默认落盘路径**：本仓库优先写入 `docs/specs/prd-<short-slug>.md`（kebab-case）；若用户指定路径或文件名，以用户为准。未经用户同意不要覆盖已有正式版本；可新建 `prd-<slug>-draft.md` 或让用户确认替换。
5. **迭代同一文件**：定稿前在同一草稿文件上增量更新，在文末维护「修订记录」表（日期、变更摘要、对话轮次可选）。

## 多轮流程（按阶段推进）

用简短状态行告知当前阶段，例如：`[PRD 阶段：澄清 - 用户与场景]`。

| 阶段 | 目的 | 代理行为 |
|------|------|----------|
| **L0 摄入** | 抓住一句话目标 | 提取已有信息：产品/模块名、谁用、要解决什么事、截止时间、是否已有设计/接口/竞品参照。 |
| **L1 澄清** | 补齐决策缺口 | 按下方「问题清单」分批提问；已明确的跳过。 |
| **L2 大纲确认** | 对齐章节与深度 | 给出文档目录 + 每节一句话意图；用户调整后再写正文。 |
| **L3 草稿** | 可评审版本 | 按模板生成完整 Markdown 写入约定路径。 |
| **L4 修订** | 吸收反馈 | 用户逐条改 → 更新对应章节 + 修订记录；同步更新「开放问题」状态（已解决/仍阻塞）。 |

若用户希望「跳过大纲直接出全文」，仍可执行，但须在回复中列出**假设清单**供用户纠正。

## 分批问题清单（按需选用，勿一轮问完）

**产品与范围**

- 文档受众是谁（研发 / 产品 / 业务 / 管理层）？是否需要中英文双语？
- MVP 边界：本期必做 vs 明确不做？
- 成功标准：上线后如何衡量「做对」（北极星指标 + 1～3 个辅助指标）？
- 硬性约束：合规、品牌、预算、排期、必须兼容的环境？

**用户与场景**

- 核心用户角色 2～4 个；每角色目标、频次、权限差异？
- 关键用户旅程 1～3 条（入口 → 成功结束态）；异常与边界？

**功能（按旅程对齐，而非功能堆砌）**

- 每个步骤：用户意图、系统响应、所需数据、权限前置条件？
- 与现有系统/页面的关系：新建、改造、替换、埋点？

**数据与集成**

- 主数据实体与来源；是否需要迁移、导入、同步频率？
- 外部系统/API：谁提供、失败降级、幂等与重试期望？

**非功能**

- 性能与容量量级（并发、数据量、延迟期望）？
- 安全与隐私（鉴权、审计日志、脱敏、多租户）？
- 可观测性：日志、指标、告警谁负责？

每轮结束附：**本轮结论摘要** + **下一轮将覆盖的主题**，邀请用户补充或纠正。

## 文档模板（写入文件时使用）

以下为章节骨架；可根据项目删减，但「验收标准」与「开放问题」建议保留。

```markdown
# <产品或功能名称> — 产品需求说明（PRD）

| 属性 | 内容 |
|------|------|
| 状态 | 草稿 / 评审中 / 已定稿 |
| 负责人 | |
| 相关链接 | 设计稿、工单、技术方案 |

## 1. 背景与问题
<!-- 为什么现在做；现状痛点；不做会怎样 -->

## 2. 目标与成功指标
<!-- 可量化或可评判的目标；验收口径 -->

## 3. 范围
### 3.1 本期包含（In Scope）
### 3.2 本期不包含（Out of Scope）

## 4. 用户与场景
### 4.1 角色概览
### 4.2 用户故事（可选表格：角色 / 我想要 / 以便于）
### 4.3 核心用户旅程
<!-- 步骤、系统行为、备注 -->

## 5. 功能需求
<!-- 建议按旅程或模块拆分；每条需求包含：描述、优先级（MoSCoW）、依赖、验收标准（Given/When/Then 或 checklist） -->

### 5.x <需求条目标题>
- **描述**：
- **优先级**：Must / Should / Could / Won't（本期）
- **验收标准**：
  - [ ] ...
- **边界与异常**：

## 6. 数据与状态（若适用）
<!-- 实体、字段级要点、状态机、校验规则；细节可链到技术文档 -->

## 7. 非功能需求
<!-- 性能、安全、兼容、可访问性、国际化、运营与支持 -->

## 8. 里程碑与依赖
<!-- MVP、后续版本；跨团队依赖 -->

## 9. 风险与开放问题
| 编号 | 问题/风险 | 影响 | owner | 状态 |
|------|-----------|------|-------|------|

## 10. 修订记录
| 日期 | 摘要 |
|------|------|
```

## 优先级与验收写法提示

- **MoSCoW** 与排期对话一致：`Must` 为 MVP 阻断项；`Won't` 写清「本期不做」以免 scope creep。
- **验收标准**尽量可测试：界面文案、状态变化、权限、数据结果、错误提示；避免「体验好」「尽量快」等不可验证表述。
- 需要拆研发任务时，在功能节末加 **「建议 Epic/Story 拆分提示」**（一句即可，不替代项目管理工具）。

## 与其他技能/流程的配合

- 若同时在做代码实现，产出 PRD 后可提醒用户：技术方案可与 `docs/specs/` 下现有 spec 交叉引用，避免重复定义接口。
- 用户若只需「一页纸」摘要：压缩为「背景 / 范围 / 用户故事+AC / 指标 / 风险」五块，仍保存为独立 md 文件以免丢失细节。

## 自检（写入文件前快速过一遍）

- [ ] 目标与指标是否可被第三方读懂并评判？
- [ ] 每条 Must 需求是否都有验收标准？
- [ ] 「不包含」是否写清楚？
- [ ] 开放问题是否都有责任人或下一步？
- [ ] 是否更新了修订记录？
