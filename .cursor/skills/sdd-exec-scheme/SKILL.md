# SDD Exec Scheme（执行方案桥接）

## 收窄原则（必须）

1. **代码现实**：只列 **本次改动会依赖、且可能需要改动或扩展** 的仓库能力（模块/类型/函数）。不列同领域但与本任务无关的条目。
2. **外部库**：写到 **本次实际依赖的包版本 + 具体 API**（导出符号、类/方法、关键类型）。无版本信息时注明「以 lockfile / 安装树为准」及核对方式。
3. **设计契约**：**从原始技术文档的设计契约出发**照抄或摘要；若与能力现实矛盾或明显不合理，**允许改写**，但必须在 Spec 中写明：**相对原文的差异**、**改写理由**（指向仓库证据）。之后 `Spec is Truth`。
4. **不留赘肉**：桥接记录只用 `references/exec-scheme-bridge.md` 的三段结构；不写长篇约束摘录表、回读清单、变更日志专节（这些交给任务 Spec 其它区块或 Light）。

## 最小工作流

- 定 **依赖边界**（本次要动的行为与文件范围）→ 填 **§1 仓库能力**、**§2 库 API** → 填 **§3 设计契约**（含对原文的继承或改写）→ 再进入 `sdd-riper-one-light`（checkpoint / 批准 / 验证）。

## 纪律

- `No Spec, No Code`：桥接未落盘不进实质编码（Light 零通道除外）。
- 发现契约与代码不一致：**先改 Spec**（含「对技术文档的改写说明」），再改代码。
