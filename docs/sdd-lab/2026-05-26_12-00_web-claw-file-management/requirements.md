# Requirements / 需求文档: Web Claw File Management

## Restated Understanding / 需求复述

- 我理解当前需求是：在 `apps/web-claw` 已有应用壳层（Files Tab 占位）、统一容器 runtime 与终端能力之上，单独梳理**文件管理**的产品范围与交互需求，为后续 `os-core` 契约与 `browserpod` 实现提供真相源。
- 当前核心目标是：定义第一阶段用户如何在 Files 区域查看容器内文件、按交互获取最新信息、读写文本与执行安全的破坏性操作；明确与 runtime session、`filePersistence` 能力声明的关系。
- 当前边界是：只定义产品与交互层面的需求，不进入 TypeScript 接口、adapter 实现、组件树或样式代码；不创建技术方案。
- 暂不处理：完整 IDE 级编辑器、Git 集成、本地上传/拖放导入、workspace 多盘/快照（`packages/web-os` FileManager 路线）、服务预览、AI 辅助、移动端完整体验。

## Context / 背景与事实

- `App.svelte` 中 Files Tab 当前为占位文案，壳层与 Tab 切换已就绪。
- BrowserPod SDK（2.8.0）公开能力包括：`createDirectory`、`createFile` / `openFile`（`TextFile` / `BinaryFile`）、`run` 的 `cwd`；**无**在类型定义中暴露的目录列表或 `fs.watch` 类 API。
- `prd-web-claw-feature-split-light.md` 阶段 2 约定：文件事实源来自**容器内命令执行结果**；不假设外部能实时监听容器内文件变化；待补 `prd-web-claw-files-light.md`。
- 用户已确认：目录信息必须通过容器内命令读取真实结果；读写文件允许在 adapter 内使用 BrowserPod SDK API。
- 用户已确认：不假设存在第三方更新源或后台实时同步；但用户每一次交互原则上都应获取当前最新信息，例如点开文件时应重新读取该文件内容。
- 用户已确认：Files 区域排版参考 VSCode / IDE 文件管理设计，左侧为可缓存的嵌套目录树，右侧为多 Tab 文件预览 / 编辑区域。
- 用户已确认：默认路径为用户目录；用户可通过类似文件管理地址栏的路径输入 / 切换能力更改路径；展示内容为当前工作区路径下的文件。
- Runtime 能力模型已有 `filePersistence: boolean`，表示 storageKey 相关持久化是否可用，不等同于「文件 UI 已实现」。
- 仓库内 `packages/web-os` 的 FileManager 面向 WebContainer + IndexedDB，**不**作为本迭代的实现真相源。

## Scope / 范围

### In

- 定义 Files Tab 内第一阶段文件管理的用户入口与排版：左侧目录树，右侧文件预览 / 编辑区域，整体参考 VSCode / IDE 文件管理设计。
- 定义容器未就绪、启动中、失败、不支持时文件区域的阻塞态与跳转容器状态面板的方式（与壳层、runtime 迭代一致）。
- 定义文件浏览的**按需同步心智**：系统不做后台实时监听；但用户进入目录、展开目录、点开文件、保存、重命名、删除等交互应触发对应范围的最新读取或更新。
- 定义目录读取的事实源：目录列表、文件/目录识别、当前路径状态必须来自容器内命令执行结果，不以 BrowserPod SDK 私有对象或前端缓存作为真相源。
- 定义目录树缓存：默认仅打开当前文件夹；已打开的目录信息由前端缓存维护，包括目录结构与展开状态（哪些路径已展开）。
- 定义路径入口：默认打开用户目录；用户可通过类似文件管理地址栏的路径输入 / 切换能力更改路径；不与终端 `cwd` 联动。
- 定义文件内容读写的实现边界：读写文件内容允许由 `browserpod` adapter 使用 BrowserPod SDK API，但 app 与 `os-core` 不暴露 SDK 类型。
- 定义打开文件模型：支持多个文件以 Tab 方式预览；当前仅支持文本文件预览、编辑和保存，其他文件类型暂不支持。
- 定义目录树上下文菜单：参考 IDE 文件树交互，为文件和文件夹提供改名、删除、新增等管理入口。
- 定义第一阶段文件操作集合的最小集合（待开放问题确认后写入验收项）：
  - 浏览：列出目录、识别文件/目录、进入子目录、返回上级。
  - 读取：查看文本文件内容（有大小/类型限制的产品规则）。
  - 写入：保存文本修改（覆盖策略与确认）。
  - 结构变更：新建文件、新建目录、重命名、删除（含确认策略）。
- 定义错误与权限类反馈：路径不存在、读写失败、容器已停止、能力不可用（如 `filePersistence` 为 false 时的说明口径）。
- 定义与 `storageKey` / 持久化的用户可见说明：变更 storageKey 即不同磁盘，不表达为外层工作区切换（继承 runtime 需求）。
- 定义 app / os-core / browserpod 的职责边界（产品层，不写接口名）。

### Out

- 不在本阶段实现代码或技术方案。
- 不设计 Portal 预览、终端 prompt、多 workspace 列表、快照导入导出。
- 不承诺实时文件监听、协同编辑、大文件流式上传的完整方案。
- 不迁移或复用 `web-os` FileManager 的 IndexedDB 多盘模型。
- 不重写容器 runtime 或终端统一交互需求，只声明依赖关系。

## User Interaction / 用户交互

### 触发入口

- 用户在 web-claw 主界面切换到 **Files** Tab。
- 容器 `running` 且文件相关 capability 可用时，文件区域可操作；否则展示统一阻塞态（文案说明依赖容器，可提供打开容器状态抽屉的入口）。

### 用户操作路径（草案）

1. 用户打开 Files Tab，左侧看到默认用户目录下的目录树，右侧看到空预览态或已打开文件 Tab。
2. 用户可通过顶部路径栏切换当前工作区路径；切换后左侧目录树展示该路径下的内容，不与终端 `cwd` 自动联动。
3. 用户展开目录时，若该目录未缓存或需要刷新，系统通过容器内命令读取该目录最新信息；已打开目录的结构与展开状态由前端缓存维护。
4. 用户点击左侧工具栏刷新目录时，系统根据缓存记录的目录树状态，依次扫描已打开的文件夹并更新缓存与展示。
5. 用户点开文本文件时，系统读取该文件当前内容，并在右侧文件预览区打开或聚焦对应 Tab；同一文件重复打开不创建重复 Tab。
6. 用户可在右侧 Tab 中预览、编辑并保存文本文件；保存按钮放置在文件预览区 Tab 行右侧，以 icon 形式存在；读取与保存文本内容可由 adapter 使用 SDK 文件 API。
7. 用户通过目录树上下文菜单对文件或文件夹执行新建、改名、删除等管理操作；操作完成后，系统更新受影响目录或文件的展示；破坏性操作前系统要求确认。
8. 操作失败时，在文件区域或 Toast 展示可读错误，不静默失败。

### Layout / 排版与区域

- 左侧目录树：
  - 顶部一行工具栏 icon 集合；第一阶段至少包含刷新目录。
  - 目录树支持嵌套展示，视觉与交互参考 IDE / VSCode 文件树。
  - 目录树维护前端缓存：已读取目录的子项、展开状态、当前选中路径。
  - 默认仅打开当前文件夹，不默认扫描全盘或全树。
- 右侧文件预览区域：
  - 包含打开文件的 Tab 管理。
  - 支持多个文本文件 Tab 的打开、切换、关闭。
  - Tab 行右侧提供文本文件保存 icon 按钮。
  - 仅支持文本文件预览、编辑、保存；其他文件类型展示不支持提示。
- 路径入口：
  - 提供类似文件管理地址栏的路径输入 / 切换能力。
  - 默认路径为用户目录；展示内容是当前工作区路径。
  - 不与终端当前目录自动联动。

### 系统反馈

- 目录读取、文件读取、刷新或保存中：展示加载态，避免重复提交同一操作。
- 非文本文件：展示暂不支持提示，不使用文本编辑器打开。
- 读取大文件：按产品规则拒绝或提示，不强行用文本编辑器打开。
- 保存成功/失败：明确反馈；覆盖已有文件时需确认。
- 容器状态变为非 `running`：禁用文件操作并说明原因。

### 不应发生的交互

- 不得在 Files Tab 内单独 boot BrowserPod 或要求用户配置 API Key（属 runtime 职责）。
- 不得把 runtime 启动失败伪装成「文件读取失败」。
- 不得在 UI 暴露 BrowserPod SDK 类型、`TextFile`/`BinaryFile` 实例或 adapter 内部对象。
- 不得默认假设存在后台实时监听；也不得要求用户必须手动刷新后才能让一次明确交互读取最新数据。

## Layer Responsibilities / 分层职责（产品）

| 层 | 职责 |
|----|------|
| **apps/web-claw** | Files Tab 布局、左侧嵌套目录树、目录树缓存与展开状态、路径栏、目录刷新工具栏、右侧多 Tab 文本预览 / 编辑、上下文菜单、操作确认 Dialog、阻塞态与 Toast；订阅 runtime 状态，不编排容器生命周期。 |
| **packages/os-core** | 定义文件服务契约：列目录、读/写文本、创建/删除/重命名、刷新状态、错误类型；接收 `RuntimeSession` 作为前置依赖。 |
| **packages/browserpod** | 基于 BrowserPod 落实文件操作；目录读取必须通过容器内命令获得真实目录信息；读写文件内容允许使用 SDK `openFile` / `createFile` 等 API；不得向 app 泄漏 SDK 类型。 |

## Constraints / 约束

- **业务**：单容器、无 workspace 切换；文件展示的是**当前容器磁盘**内容。
- **技术**：遵守 `src/lib` 分层；文件 UI 在 `features/files`（或等价域）；编排逻辑用类封装于 `core` / `packages`，组件尽量只展示与转发意图。
- **安全**：不提交 API Key、`.env`、用户文件内容到仓库。
- **设计**：视觉与交互实现阶段以 `docs/design/BitsUI.md` 为准。
- **兼容**：第一阶段以桌面 Chromium 系浏览器为验证基线。
