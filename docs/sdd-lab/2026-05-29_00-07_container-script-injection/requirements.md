# Requirements / 需求文档: Container Script Injection

## Restated Understanding / 需求复述

- 我理解当前需求是：为容器运行时提供一套公共的脚本注入契约，并由具体平台实现该契约；首个实现目标是 BrowserPod。
- 当前核心目标是：让容器启动完成后，注入的 alias 指令处于可用状态；提前注入或启动后注入属于具体平台实现阶段根据底层能力选择的技术路径。
- 当前边界是：先定义需求，不进入代码实现；需求契约不关心具体注入时机，只要求容器启动后 alias 指令可运行、可维护、可幂等注入。
- 暂不处理：不替换具体运行时的底层系统工具本身；不承诺注入脚本覆盖完整 GNU 工具行为；不在需求阶段确定具体 API、文件结构或执行时序。

## Scope / 范围

- In:
  - 定义容器脚本注入的公共能力契约，供不同容器运行时实现。
  - 支持具体实现选择合适的容器内受管目录，用于保存注入脚本和相关 meta 信息。
  - 支持以正常 JS 文件形式维护注入脚本，并提供专门的开发目录管理脚本文件。
  - 支持管理注入脚本本体及其相关 meta 信息，例如名称、版本、来源文件、更新时间、启用状态或校验信息。
  - 当前阶段仅支持通过 shell 配置文件注入 alias 指令，不支持 PATH/bin wrapper、默认 env 注入或直接替换系统命令。
  - 支持通过目标容器可用的脚本运行器执行注入脚本；BrowserPod 首选 Node。
  - 注入行为必须幂等，重复调用不应产生重复 alias、重复脚本副本、重复 meta 项或破坏用户已有配置。
  - 支持在容器启动后获得可用注入指令；提前注入或启动后注入由技术方案阶段根据具体运行时底层能力确认。
- Out:
  - 不要求修改具体运行时官方实现或底层镜像。
  - 不要求让所有 shell 都完全一致地加载配置，只要求明确支持目标 shell 和加载文件。
  - 不要求注入脚本永久跨所有 storage/session 存活；持久化策略由技术方案阶段确认。
  - 不处理通用插件市场、远程脚本下载、权限沙箱策略等扩展能力。
  - 当前阶段不要求注入命令在非交互式 `sh -c`、文件命令层或没有加载 profile 的终端模型中可直接解析。

## User Interaction / 用户交互

- 触发入口：
  - 应用在创建或恢复容器会话时触发注入。
  - 后续可提供手动“重新注入/修复注入”的维护入口。
- 用户操作路径：
  - 用户启动容器或打开终端。
  - 系统确保注入所需脚本、meta 和 shell 配置片段以可控形式存在。
  - 用户在已加载对应 shell 配置的终端内执行被注入的 alias 指令。
- 系统反馈：
  - 注入成功时应能提供可追踪状态，例如已注入、无需变更、已修复或失败原因。
  - 重复注入时应反馈为无变更或幂等完成。
- 状态变化：
  - 未注入 -> 已注入。
  - 已注入 -> 无变更。
  - 已注入但内容过期 -> 已更新。
  - 已注入但 profile 片段缺失 -> 已修复。
- 异常/边界交互：
  - profile 文件不存在时应创建或追加必要内容。
  - profile 已存在用户内容时不得覆盖用户配置。
  - 目标目录存在但 meta 不完整时应可恢复。
  - 容器缺少 Node 或 shell 加载行为不符合预期时应返回明确失败或降级说明；不通过 PATH/bin wrapper 兜底。
  - 注入指令应支持参数控制是否强制覆盖既有注入内容；默认不探测、不覆盖用户自定义修改。
- 不应发生的交互：
  - 重复调用后出现多段重复 alias。
  - 删除或重写用户已有 profile 内容。
  - 注入失败但界面仍报告成功。

## Acceptance Criteria / 验收标准

- [ ] 容器用户目录中存在稳定的隐藏注入目录，用于存放脚本和 meta 信息。
- [ ] 开发态注入脚本以正常 JS 文件存在，避免把脚本内容维护为大段字符串。
- [ ] 注入脚本可通过 Node 执行，并能通过 alias 指令被调用。
- [ ] `~/.profile` 或最终确认的 shell 配置文件中存在可识别、可更新、可移除的注入片段。
- [ ] 重复执行注入不会产生重复配置、重复脚本副本或重复 meta 项。
- [ ] 注入逻辑不会覆盖用户已有 profile 内容。
- [ ] 注入指令支持通过参数显式请求强制覆盖；未传入强制参数时不得覆盖用户自定义内容。
- [ ] 注入结果能区分“首次注入”“无需变更”“已更新/已修复”“失败”。
- [ ] 无论实现采用启动前注入还是启动后注入，容器启动完成后，在目标 shell 加载对应配置后 alias 指令应可运行。
- [ ] 对中文文件名 `ls` 显示异常这类工具替代需求，注入方案能够承载对应 Node 脚本，但不把具体脚本功能绑定为本需求唯一目标。

## Constraints / 约束

- 业务约束：
  - 注入能力应服务于容器用户体验增强，不应让用户误以为底层系统工具已被官方修复。
  - 用户已有配置优先，系统注入内容必须可识别、可幂等维护。
- 技术约束：
  - 公共契约不绑定 BrowserPod API；BrowserPod 的启动前写入、`pod.run`、文件 API 等能力只属于具体实现事实。
  - 目标配置文件应尽可能跨终端支持；具体选择 `~/.profile`、`~/.bashrc`、`ENV` 或组合策略由技术方案阶段确认。
  - 终端粘贴/heredoc 在 BrowserPod 中可能不稳定，应优先考虑通过文件 API 或受控写入路径生成脚本。
- 时间/兼容性约束：
  - 先覆盖 BrowserPod 当前接入版本，再评估后续版本兼容。
  - 不在需求阶段承诺兼容全部 POSIX shell 启动模式。

## Open Questions / 开放问题

- [x] 需求契约是否要求指定启动前注入或启动后注入？
  - 结论：不要求。契约只要求容器启动后注入指令可用；具体注入时机由技术方案阶段基于底层支持决定。
- [x] shell 配置文件是否固定为 `~/.profile`？
  - 结论：不固定。目标是尽可能跨终端支持，具体配置文件或组合策略由技术方案阶段确认。
- [x] meta 和脚本的开发态/注入态形态是否在需求阶段固定？
  - 结论：不固定。需求只要求开发时可拓展可控，最终注入后指令可运行；存在形式由技术方案阶段设计。
- [x] 指令暴露方式是否固定为 alias？
  - 结论：当前阶段固定为 alias 注入指令；不再评估 `~/bin` 可执行文件、PATH wrapper 或默认 env 注入作为本轮方案范围。
- [x] 是否自动探测用户修改并覆盖？
  - 结论：不探测；注入指令通过参数控制是否强制覆盖，默认不得覆盖用户自定义内容。
- [x] 公共契约是否应该绑定 BrowserPod？
  - 结论：不应该。契约是公用的，实现才针对具体平台；BrowserPod 是首个实现目标。
- [x] 注入脚本是否用 `content` 字符串维护？
  - 结论：不应该。脚本应以正常 JS 文件存在，并由专门开发目录管理；契约和实现只引用脚本资产，不把大段脚本内容内联进配置。
