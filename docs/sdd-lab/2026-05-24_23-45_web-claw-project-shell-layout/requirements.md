# Requirements / 需求文档: Web Claw Project Shell And Layout

## Restated Understanding / 需求复述

- 我理解当前需求是：现有需求已覆盖 runtime、终端等局部能力，但缺少 web-claw 应用本身如何搭建，以及整个系统页面如何组织的顶层需求。
- 当前核心目标是：先定义 web-claw 的最小项目壳层、页面布局、导航结构、功能区域和状态承载方式，为后续容器 runtime、终端、文件、预览等能力接入提供统一应用框架。
- 当前边界是：只定义产品与交互层面的需求，不进入具体技术选型、包接口、组件实现或样式代码。
- 暂不处理：具体 runtime adapter 实现、终端输入输出细节、文件管理细节、Portal 预览细节、未来容器内部工作区能力、生产部署方案。

## Scope / 范围

- In:
  - 定义 web-claw 项目搭建的第一阶段目标和最小可用边界。
  - 定义系统级页面布局：应用外壳、顶部/侧边/主区域/辅助面板等区域职责。
  - 定义核心功能入口：容器状态、终端、文件、预览、任务/日志等能力如何在页面上被组织。
  - 定义容器未就绪、启动中、失败、不支持等基础设施状态在系统页面中的承载方式。
  - 定义页面布局对后续功能迭代的插槽和扩展原则。
  - 定义与现有 PRD、终端统一交互、容器 runtime 管理迭代之间的关系，并明确旧 `docs/specs` 作为禁用参考，不再作为实现真相源。
- Out:
  - 不在本阶段实现代码。
  - 不在需求阶段确定具体 Svelte 组件树、TypeScript 接口或状态管理实现。
  - 不设计完整文件管理、Portal 预览、命令历史、AI 辅助或账号系统。
  - 不覆盖移动端完整体验；第一阶段以桌面浏览器为主。
  - 不重写已确认的容器 runtime 管理或终端统一交互需求，只定义它们在系统页面中的承载位置。

## User Interaction / 用户交互

- 触发入口：
  - 用户打开 web-claw 应用。
  - 用户进入页面后使用唯一当前容器；第一阶段不提供工作区列表、创建、选择或切换。
  - 用户通过系统页面访问终端、文件、预览或容器状态。
- 用户操作路径：
  - 用户进入应用后看到清晰的应用壳层，而不是直接落到单一功能 demo。
  - 用户进入页面后系统自动触发唯一当前容器启动；容器准备期间，主区域展示终端加载态。
  - 容器准备完成后，主区域优先展示终端。
  - 用户可识别当前容器状态、主要工作区域和可用工具。
  - 用户可从抽屉详情查看容器状态，并理解终端、文件、预览为什么可用或不可用。
  - 用户可在同一个系统框架内通过 Tab 切换终端、文件、预览等能力；未实现功能先展示占位态。
  - 用户可通过 Tab 右侧的更多菜单访问容器关机、打开容器面板等系统操作。
- 系统反馈：
  - 当项目尚未连接 runtime 时，页面展示基础设施未就绪的阻塞态。
  - 当容器启动中时，页面展示终端加载态和必要的全局启动状态，避免各功能重复报错。
  - 当容器失败或不支持时，页面提供统一错误摘要与下一步建议。
  - 当功能尚未实现时，页面应展示明确占位态，而不是空白区域。
- 状态变化：
  - 页面布局本身不定义新的 runtime 状态，只消费容器 runtime 管理迭代定义的状态。
  - 页面需要能根据 runtime session 和功能可用性更新各区域启用状态。
- 异常/边界交互：
  - 若 runtime 不可用，终端、文件、预览等区域不应各自触发独立 boot。
  - 若某个功能尚未接入，页面应保留结构和占位说明，不阻塞其他功能。
  - 若屏幕宽度不足，页面应优先保留主区域和关键状态入口。
  - 若用户触发容器关机，页面应进入统一容器停止语义，不拆分多个底层停止概念。
- 不应发生的交互：
  - 不应让 web-claw 只表现为 BrowserPod 或终端 demo。
  - 不应把容器错误散落在各个功能模块里重复展示。
  - 不应在页面布局中暴露 API Key、BrowserPod SDK 实例或底层 adapter 细节。
  - 不应在第一阶段引入独立工作区概念、多容器或容器切换管理。

## Project Setup Requirements / 项目搭建需求

- web-claw 第一阶段应是可承载真实功能迭代的应用项目，而不是一次性 demo。
- web-claw 第一阶段应创建正式 app，而不是继续把 demo 作为系统入口。
- web-claw 第一阶段只允许一个当前容器，进入页面即启动；该容器可理解为一台计算机。
- 第一阶段不引入独立 workspace 概念；如果未来存在工作区，也应作为容器内部的能力处理，而不是系统页面外层的上下文。
- 项目搭建应明确 app、共享 UI、核心能力包、runtime adapter、feature 模块之间的边界。
- 项目搭建应支持后续接入：
  - 容器 runtime 管理。
  - 统一终端交互。
  - 文件管理。
  - 服务预览。
  - session 管理。
- 项目搭建应保留 demo 与正式 app 的边界，避免 demo 代码直接成为系统壳层。
- 项目搭建应遵守仓库既有 `src/lib`、`packages` 和 Svelte/Vite 组织约定。
- 容器逻辑编排应采用三层架构：
  - 纯 TS 编排层：负责容器启动、停止、重试、事件订阅和状态转换，不依赖 Svelte 组件或 Svelte `$state`。
  - store 状态层：负责持有可订阅状态并驱动页面更新。
  - 展示组件层：尽可能只展示状态、触发用户意图，不承载容器生命周期编排。
- 容器编排实例应作为项目级单例注入到 app 中，避免多个页面组件各自创建 runtime manager 或重复 boot。
- 旧 `docs/specs` 可作为历史背景读取，但应标记为禁用参考；后续实现以本 SDD Lab 需求、技术方案和已确认迭代为准。

## Page Layout Requirements / 页面布局需求

- 系统页面应至少包含：
  - Tab 顶部区域：承载终端、文件、预览等能力入口，以及当前容器状态入口和更多菜单。
  - 主区域：默认承载终端视图；容器准备前展示终端加载态。
  - Tab 区域：承载终端、文件、预览等可切换能力；未实现的文件和预览先展示占位态。
  - Tab 右侧更多菜单：承载容器关机、打开容器面板等系统操作入口。
  - 抽屉详情：承载容器状态、能力摘要、日志、问题提示或详情面板。
  - 状态与占位层：承载未启动、启动中、失败、不支持、未实现等状态。
- 页面布局应支持终端、文件、预览的 Tab 切换组织，第一阶段不要求并排完整实现。
- 第一阶段不需要单独 Header；产品名、容器状态和全局操作如需出现，应收敛在 Tab 顶部区域或 More 菜单中。
- 容器状态面板应作为系统级抽屉详情存在，并能被终端、文件、预览等阻塞态打开或聚焦。
- 页面视觉与交互后续实现应以 `docs/design/BitsUI.md` 为设计约束来源。
- 第一阶段页面应优先保证信息架构清楚、状态可解释、扩展位置稳定，再追求复杂动效或完整视觉 polish。

## Constraints / 约束

- 业务约束：
  - 用户需要先理解当前容器和基础设施状态，再使用终端、文件、预览等能力。
  - 系统页面应能承载逐步接入的功能，不能要求所有功能一次性完成后才可用。
  - 页面应避免把基础设施问题误表达为某个功能自身失败。
- 技术约束：
  - 项目结构应遵守仓库既有 `src/lib` 与 `packages` 约定。
  - UI 与业务逻辑应分离，公共 UI 与业务组件应分离。
  - 容器生命周期编排必须与 Svelte 组件分离，不得把 boot、stop、retry、event wiring 写入展示组件。
  - 容器核心状态不得用 Svelte `$state` 作为真相源；页面更新应通过 store 订阅获得。
  - app 不应直接 import BrowserPod SDK 类型或暴露 adapter 细节。
  - 旧 `docs/specs` 与本 SDD Lab 文档冲突时，以本 SDD Lab 文档为准。
  - API Key、`.env`、本地 runtime 数据和机器路径不得进入提交内容。
- 时间/兼容性约束：
  - 第一阶段以桌面 Chromium 系浏览器为验证基线。
  - 第一阶段允许功能占位，但必须明确状态和后续扩展位置。
  - 移动端、账号体系、云端同步、完整部署不作为本轮必要范围。
