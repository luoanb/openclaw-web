# Requirements / 需求文档: Web Claw Service Preview

## Restated Understanding / 需求复述

- 我理解当前需求是：当容器内启动开发服务或其他网络服务时，服务会暴露一个可访问的网络地址；web-claw 需要把这个地址作为“服务预览”能力呈现给用户。
- 当前“预览”不是文件内容预览，而是对容器内运行服务的页面或接口地址进行发现、展示、打开、刷新和失败反馈。
- 当前核心目标是：定义服务地址从用户视角如何出现、如何进入 Preview Tab、多个地址如何组织，以及服务不可用时如何解释。
- 当前边界是：先定义产品与交互需求；不在本阶段决定具体代理、端口映射、地址解析、iframe sandbox、BrowserPod SDK API 或 TypeScript 接口。
- 暂不处理：文件预览、移动端完整体验、生产发布域名策略、鉴权服务预览、跨容器多服务管理、后端持久化历史记录。

## Related Iterations / 关联迭代

- `2026-05-24_23-45_web-claw-project-shell-layout`：已将预览作为 web-claw 顶层 Tab 能力之一，本迭代细化 Preview Tab 的服务预览语义。
- `2026-05-24_21-16_container-runtime-management`：服务预览依赖当前唯一容器处于可运行状态，但不重新定义 runtime 状态。
- `2026-05-24_20-38_terminal-unified-interaction`：服务地址可能来自终端命令输出或容器事件；本迭代只定义用户可见需求，不扩展终端公共契约。
- `2026-05-26_12-00_web-claw-file-management`：文件预览与服务预览是不同功能面，不应混用文案或状态。

## Scope / 范围

### In

- 定义服务预览的对象：容器内启动的网络服务地址，例如 dev server、web app、文档站点或本地 API 地址。
- 定义用户可见入口：
  - Preview Tab 应承载当前可预览服务。
  - 当系统识别到服务地址时，应让用户能明确打开或切换到该预览。
  - 用户也应能在 Preview Tab 中手动输入或粘贴一个服务地址作为兜底入口。
- 定义服务地址展示：
  - 展示服务地址本身或可理解的服务名称。
  - 多个服务地址同时存在时，应提供可切换列表，而不是只保留最后一个。
  - 最近发现或最近打开的服务应更容易继续访问。
- 定义预览行为：
  - 打开服务地址后，在 Preview 区展示该服务页面。
  - 用户可刷新当前预览。
  - 用户可在新浏览器标签页打开当前服务地址。
  - 用户可清除不可用或不再需要的地址记录。
- 定义状态与反馈：
  - 容器未运行时，Preview Tab 展示阻塞态，并引导用户查看容器状态。
  - 尚未发现服务地址时，Preview Tab 展示空态，并提示用户先在终端启动服务或手动输入地址。
  - 服务加载失败、连接失败、地址不可访问或浏览器阻止嵌入时，应展示可读错误与可执行下一步。

### Out

- 不把文件内容预览纳入本迭代。
- 不承诺自动识别所有框架输出格式；识别规则留到技术方案阶段基于项目能力确认。
- 不承诺跨会话持久保存服务地址历史；是否持久化留待技术方案阶段确认。
- 不承诺在需求阶段选择 iframe、独立窗口、代理 URL 或 BrowserPod 官方预览 API。
- 不在本迭代设计服务端鉴权、cookie 隔离、HTTPS 证书策略或公网分享能力。

## User Interaction / 用户交互

### 触发入口

- 用户在 Terminal Tab 中启动网络服务，服务输出可访问地址。
- 系统从终端输出、runtime 事件或后续可用的容器能力中识别到服务地址。
- 用户进入 Preview Tab，查看已发现服务地址列表或空态。
- 用户在 Preview Tab 手动输入一个服务地址。

### 用户操作路径

1. 用户启动服务后，系统识别到服务地址并在 Preview Tab 中记录该地址。
2. 若用户当前不在 Preview Tab，系统应以轻量方式提示有可预览服务可打开，不能强行切走用户当前工作区。
3. 用户进入 Preview Tab 后，可看到一个或多个服务地址。
4. 用户选择某个地址，Preview 区加载对应服务页面。
5. 用户可刷新当前预览，或在新浏览器标签页打开。
6. 用户可手动添加地址；添加后地址进入同一服务列表并可被打开。
7. 当服务不可访问时，Preview 区显示错误说明，并保留地址，方便用户修复服务后刷新。

### 系统反馈

- 服务地址被发现时，应给出可感知但不打断的提示。
- Preview Tab 空态应说明“先启动服务或手动输入地址”，而不是只显示空白。
- 加载中应有明确状态，避免用户误以为页面卡死。
- 加载失败时应区分“地址格式不支持”、“服务未响应”、“容器未运行”、“浏览器无法嵌入或跨源限制”等用户可理解类型。
- 多个服务地址存在时，当前选中项应清晰可见。

### 不应发生的交互

- 不得把文件预览和服务预览混成同一个需求语义。
- 不得在发现服务地址时自动中断用户当前终端或文件操作。
- 不得只展示原始错误堆栈而没有用户可理解的下一步。
- 不得因容器未运行而让 Preview Tab 自行触发独立 boot；容器生命周期仍由 runtime 管理能力承载。
- 不得把服务地址中的敏感 token 写入可提交文档、测试快照或日志样例。

- [x] 需求文档明确服务地址可能由终端输出、runtime 事件或手动输入产生。

- [x] 需求文档明确多个服务地址需要可切换列表。
- [x] 需求文档明确用户可刷新当前预览并在新浏览器标签页打开。
- [x] 需求文档明确容器未运行、无地址、加载中、加载失败等状态需要用户可见反馈。

- [x] 技术方案需确认 BrowserPod 是否已有官方服务预览、端口映射或 URL 转换能力。

## Constraints / 约束

- 产品约束：第一阶段仍是单当前容器语义；Preview Tab 只展示当前容器内服务，不处理多容器服务聚合。
- 交互约束：服务地址发现提示必须轻量，不抢占用户正在进行的终端、文件或编辑操作。
- 技术约束：Preview 功能不得绕过 `os-core` / adapter 分层直接把 BrowserPod SDK 细节暴露给 app 组件。
- 状态约束：Preview 只消费 runtime 状态，不定义新的容器生命周期状态。
- 安全约束：服务 URL、查询参数和错误日志需要考虑脱敏；不得把包含 token 的真实地址写入仓库示例。
- 兼容约束：第一轮验证以桌面 Chromium 系浏览器为基线。

## Open Questions / 开放问题

- [x] 服务地址的主要发现来源是什么？
  - 结论：第一阶段自动发现主路径采用 BrowserPod 官方 Portal / `onPortal` 事件；手动 URL 输入作为兜底；终端输出解析留作后续增强。
- [x] 终端输出识别是否需要第一阶段覆盖常见 localhost / 0.0.0.0 / 127.0.0.1 / LAN URL 格式？
  - 结论：第一阶段不做终端输出识别。原因是当前 BrowserPod default terminal 输出未稳定进入 `TerminalEvent.terminal-output`，不应解析 SDK DOM。
- [x] 多个服务地址的排序规则是什么？
  - 结论：第一阶段当前选中项优先，其次按最近打开时间倒序，再按发现时间倒序；后续可再做用户置顶。
- [x] 服务地址是否需要跨页面刷新保留？
  - 结论：第一阶段不跨会话持久化，保留在当前 runtime session 的内存状态中。
- [x] Preview 区默认使用内嵌页面、外部新标签页，还是两者同时提供？
  - 结论：两者同时提供；默认 iframe 内嵌预览，同时提供外部新标签页打开。
- [x] 当服务需要 WebSocket、SSE 或热更新时，预览承载方式是否有额外限制？
  - 结论：第一阶段不单独实现 WebSocket/SSE 特殊逻辑，默认依赖 BrowserPod Portal URL 与 iframe/外部打开；若真实验证发现限制，再先回写技术方案。
