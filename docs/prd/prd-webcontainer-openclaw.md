## 1. 背景与问题

`demos/webcontainer-openclaw` 当前已完成 **WebContainer 内安装 `openclaw` 并执行 CLI 探测** 的 PoC，输出为只读日志流、单按钮流程。若要在浏览器内更贴近「本地开发体验」，需要：

- **可管理的终端式交互**：不仅执行固定脚本，还要能发起、观察、（在能力范围内）控制容器内命令，以便排障、尝试子命令、运行简单 dev 任务。
- **可预览容器内 HTTP 服务**：当用户在容器内启动 Web 服务（如静态站、Vite dev、OpenClaw 相关 HTTP 能力若可用）时，宿主页面需能 **展示该服务页面**，而不是仅看 stdout。

不做则 demo 停留在「一键跑通 + 打日志」，无法支撑「在沙箱里试跑一个带 UI 的进程」类场景，与产品化体验差距大。

## 2. 目标与成功指标

| 目标 | 说明 | 建议指标（可再定） |
|------|------|-------------------|
| 终端可管理 | 用户可在 PoC 页面内对容器内命令进行「发起 / 查看输出 / 基本生命周期管理」 | 核心路径可用率（本地手测清单全绿）；关键错误有可读提示 |
| 服务可预览 | 容器内监听端口的服务可被宿主以 iframe 或等价方式展示 | 常见 dev server 场景下能打开预览 URL 并可见页面加载 |

**北极星（可选）**：在支持跨源隔离的浏览器中，用户无需离开 demo 页即可完成「执行命令 → 看到终端输出 →（若适用）打开容器内 Web 页」闭环。

## 3. 范围

### 3.1 本期包含（In Scope）

- **宿主布局与信息架构（Must）**
  - **顶部 Tab**：两个 Tab — **终端** / **预览**，用于切换主工作区。
  - **无品牌/标题栏**：页面 **不提供** 独立品牌条或主标题栏（见 §9.1 q5）；仅保留 **Tab 行**，以及 **必要时** 与 Tab 同龄或内联的 **极简系统状态**（如 Boot / `crossOriginIsolated`、预览错误），外链预览与帮助可收入 **图标按钮 / 溢出菜单 / 页脚一行**。
  - **全屏工作区**：除 Tab 条（及上述极简状态占用高度）外，**当前 Tab 对应面板铺满可视区域**（`100vh` / `100dvh` 语义，扣减 Tab 高度），无大块 PoC 装饰区占位。
  - **页面精简（生产观感 Must）**：去掉或减少 PoC 向的长说明横幅；长文案降级为折叠、帮助链接或开发环境可见。
  - **Tab 切换时保持挂载（已决策）**：切换到另一 Tab 时 **不卸载** 预览 iframe、**不中断** 终端侧已运行进程（用户未主动中止前）；切回后立即可见（见 §9.1 q4）。

- **终端指令管理（Must）**
  - **VS Code 终端式体验**：以 **xterm.js**（及 fit 等 addon）呈现终端；**命令输入仅在 xterm 内完成**（已决策，见 §9.1 q3），不提供独立命令输入框作为主路径。
  - **命令解析（已决策）**：用户每行输入作为 **整行** 交给容器内 shell 执行（如 `sh -c '<整行>'`），具体转义与安全边界在实现 Spec 中定义（见 §9.1 q1）。
  - **交互式 stdin / REPL（已决策，Must 包含）**：本期 **必须** 覆盖需标准输入交互的场景（如 REPL、交互提示），在 `@webcontainer/api` 能力范围内接好 **stdin 泵**；**不承诺** 与本地真 PTY 行为逐字节一致，无法实现时在帮助文档中说明差异（见 §9.1 q2）。
  - **单会话模型（已决策）**：同一时刻 **单前台 shell/命令会话**（无多标签多会话并行）；新命令策略（排队 / 拒绝 / 先中止）在实现 Spec 中写清（见 §9.1 q7）。
  - **输出展示**：stdout/stderr 流式写入终端；**日志截断**见 §6、§7（§9.1 q8）。
  - **基本管理**：退出码可见；运行中进程 **中止**（若 SDK 允许）。
  - **可选增强（Should）**：最近若干条命令历史、清空缓冲、预设命令快捷（如 `npx openclaw --help`）。

- **容器内 Web 服务展示（Must）**
  - 当容器内进程在约定端口提供 HTTP 服务时，宿主能够 **获得可访问的预览 URL** 并在 **「预览」Tab** 下 **内嵌展示**（iframe 或 WebContainers 官方 `preview` 能力，技术选型见 §9.2 T1）。
  - 用户能 **看到服务已就绪的提示**（含端口/链接）；**非选中 Tab 时保持挂载**（§9.1 q4）。

### 3.2 本期不包含（Out of Scope）

- 完整 OpenClaw **网关、全通道、真实 LLM 代理**（与 [可行性研究](../research/feasibility-openclaw-webcontainers.md) 及现有 PoC Spec 一致）。
- 修改 `demos/openclaw` 源码；生产级多租户、登录鉴权、审计。
- 在 WebContainer 内复刻整仓 monorepo workspace（除非单独决策）。
- **与本地操作系统级 TTY/PTY 完全等价**（信号、伪终端全特性、所有 curses 场景）；本期目标为在浏览器与 WebContainer 约束下 **尽力满足 stdin/stdout 交互与常见 CLI**，差距以文档说明。
- **显式多会话并行 UI**（多 Tab 多 shell）；本期为 **单会话**（§9.1 q7）。

## 4. 用户与场景

### 4.1 角色概览

| 角色 | 目标 |
|------|------|
| 内部研发 / 技术产品 | 验证 OpenClaw 在 WC 内的行为、向他人演示 |
| 社区或贡献者 | 在浏览器沙箱中尝试命令，减少本机环境差异 |

### 4.2 用户故事

| 角色 | 我想要 | 以便于 |
|------|--------|--------|
| 研发 | 在页面里输入并执行容器内命令，并看到完整输出 | 排障和尝试不同 CLI 参数 |
| 研发 | 在容器里启动一个 web 服务后，在同一页看到页面效果 | 验证 HTTP 层行为而不仅看日志 |
| 研发 | 停止长时间运行的命令或重新执行 | 避免卡死或重复刷新整页 |

### 4.3 核心用户旅程

1. 用户打开 demo（`crossOriginIsolated` 就绪）→ WebContainer 已 boot（可与现有一致：首次操作再 boot）。
2. 用户在 **终端** Tab 下以类 VS Code 方式输入命令（如 `npm run dev`）→ 输出流式显示 → 可中止（若支持）。
3. 若命令启动了监听端口的服务 → 用户切换到 **预览** Tab → 全幅加载容器内 URL（或官方 preview URL），用户可见页面。
4. 异常：命令失败、端口冲突、预览跨域 → 界面展示明确错误与指向可行性/PoC 文档的说明。

## 5. 功能需求

### 5.0 整体布局（ Tab + 全屏，示意）

以下为 **单面板全屏** 信息架构草图（Tab 条以下全部为当前 Tab 内容区）。**无独立品牌/标题栏**；实现可与下图等价，允许微调 Tab 高度与状态展示形式。

```text
┌──────────────────────────────────────────────────────────────────┐
│  [ 终端 ]    [ 预览 ]     （无品牌/主标题栏；可选内联状态 · Boot / COI） │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│              当前 Tab 面板：高度 = 视口 − Tab 条                  │
│                                                                  │
│   · 终端 Tab：xterm 容器铺满；输入以「类 VS Code 终端」方式交互   │
│   · 预览 Tab：iframe（或官方 preview）铺满；工具条可置顶一行      │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

- **优先级**：Must
- **验收标准**：
  - [ ] 用户可通过顶部 Tab 在 **终端** 与 **预览** 之间切换，切换后对应面板占满主内容区（无大面积空白「占位卡片」）；**切换 Tab 不中断** 终端内未中止进程、**不卸载** 预览 iframe（§9.1 q4）。
  - [ ] **无** 独立品牌条或页面主标题栏（§9.1 q5）；页面首屏以 **工作区为主**。
  - [ ] 根容器在常见桌面浏览器下 **铺满窗口**；**移动端/窄屏** 以实现可行优先，不强制单独里程碑（§9.1 q6）。
- **边界与异常**：极小视口下 Tab 与终端可读性；若需保留「并排分栏」作为可选布局，单独 Story，不作为本期默认。

### 5.1 终端指令管理

- **描述**：提供面向 WebContainer 的命令执行与输出管理能力，替代「仅单按钮固定流水线」；**交互目标**为 VS Code 终端级：**xterm 内完整键盘与选区体验** + 与 WC `spawn` 输出泵对接；保留 PoC 一键探测可作为命令面板/预设（Should）。
- **优先级**：Must
- **验收标准**：
  - [ ] 终端区域获得焦点后，用户在 **xterm 内** 输入 **整行命令** 并执行（§9.1 q1、q3）；无独立命令输入框作为主路径。
  - [ ] **交互式 stdin**（§9.1 q2）：在能力范围内，子进程可从用户输入读取 stdin；不支持的场景在帮助中列出。
  - [ ] stdout/stderr 流式写入 xterm；**超长日志按配置文件截断**（§9.1 q8），默认值在配置文件中给出「推荐默认」并可在实现 Spec 中引用。
  - [ ] 执行结束显示退出码；运行中可中止（若当前 `@webcontainer/api` 对应该进程可用）。
  - [ ] 支持复制选区、粘贴等与 xterm 默认一致的行为。
  - [ ] **单会话**（§9.1 q7）：界面或帮助中说明「同时仅一条前台会话」及发起第二条命令时的行为。
- **边界与异常**：命令为空、boot 失败、`crossOriginIsolated` 为 false 时提示明确；与 **完整 PTY** 的差异见 §3.2。
- **建议 Epic/Story 拆分提示**：xterm 挂载与 fit；spawn 泵；stdin 策略；历史/中止。

### 5.2 容器内 Web 服务预览

- **描述**：识别或配置容器内 HTTP 服务地址；在 **「预览」Tab** 内全屏（扣工具条）嵌套展示。
- **优先级**：Must
- **验收标准**：
  - [ ] 在典型场景（如 `server-ready` 或等价事件）下，服务启动后用户切换到 **预览** Tab 可 **全幅** 查看 iframe（或官方 preview）内容，并有刷新/外开等至少一种可用操作。
  - [ ] 若服务未启动或 URL 无效，预览区显示占位或错误状态，而非白屏无提示。
- **边界与异常**：混合内容、COOP/CEP 与 WebContainer 文档中关于 preview 的限制；若必须新窗口打开，需在 PRD 修订中说明并更新验收。
- **建议 Epic/Story 拆分提示**：端口/URL 发现与「预览壳」UI 分 story；保持挂载策略已验收（§9.1 q4）。

### 5.3 PoC 回归与兼容

- **描述**：不破坏现有「安装 openclaw + `--help`」能力；可作为预设命令或首屏引导保留。
- **优先级**：Should
- **验收标准**：
  - [ ] 仓库根 `pnpm --filter webcontainer-openclaw` 的 build/tsc 仍通过 CI 或本地脚本约定。
- **边界与异常**：无。

## 6. 数据与状态（若适用）

- **会话状态**：WebContainer 单例策略保持与现有一致（每页一次 `boot`）；终端 **单会话**（§9.1 q7）。
- **终端状态**：当前进程引用、输出缓冲区；**日志截断**由 **单一配置文件** 管理（如项目根或 `src/` 下约定文件名），内含 **推荐默认**（行数或字节上限、是否仅保留尾部等，实现 Spec 写死字段名与默认值）；运行时可读取环境或构建时注入，但 **产品层面** 以「一份配置 + 合理默认」为准（§9.1 q8）。
- **预览状态**：当前预览 URL、加载状态（loading / error / ready）；切换 Tab **保持挂载**（§9.1 q4）。

## 7. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | **日志截断**：通过 **单一配置文件** 设定上限（总行数、总字节或单行长度等）及 **推荐默认值**；超出策略（丢弃头部仅保留尾部 / 停止追加等）在实现 Spec 中定义，避免主线程长时间阻塞（§9.1 q8）。 |
| 响应式 | **移动端/窄屏**：以实现可行与桌面优先为准，不强制单独交付里程碑（§9.1 q6）。 |
| 安全 | 运行于用户浏览器沙箱；不向宿主持久化敏感令牌；预览 URL 仅来自 WC 提供的事件或用户明确配置的端口。 |
| 兼容 | 与 Vite COOP/COEP 配置兼容；目标浏览器为官方支持的 Chromium 系为主。 |
| 可访问性 | 终端区与预览区有语义或可聚焦区域；关键操作用 `aria-live` 或等价提示（延续现有 `aria-live` 思路）。 |

## 9. 风险与开放问题

### 9.1 产品决策摘要（用户确认，自上而下 q1→q8）

| 项 | 用户原话 | 产品结论 |
|----|-----------|-----------|
| q1 | 整行 | 每行命令 **整行** 提交容器内 shell（如 `sh -c`）；转义与安全细则归实现 Spec。 |
| q2 | 包含 | **交互式 stdin / REPL** 纳入本期 Must；在 WC/API 范围内接 stdin，与真 PTY 差异见 §3.2 与帮助文档。 |
| q3 | xterm 内输入 | **仅 xterm 内输入** 为主路径，无独立命令输入框。 |
| q4 | 挂载，不中断运行 | 切换 Tab **保持挂载**；**不中断** 终端内仍在运行的进程（用户未中止前）；预览 iframe 不卸载。 |
| q5 | 不需要品牌和标题栏 | **无** 品牌条/主标题栏；仅 Tab + 可选极简系统状态；外链/帮助归图标或溢出菜单等（实现定）。 |
| q6 | 都行，能实现为主 | **移动端/响应式** 不强制单独里程碑，**以实现可行、桌面优先**。 |
| q7 | 单会话 | **单会话** UI：同时仅一条前台 shell/命令会话；第二条命令行为由实现 Spec 定义。 |
| q8 | 日志可截断（配置文件 + 最佳默认值） | **单一配置文件** 管理截断规则与 **推荐默认**；字段与默认值在实现 Spec 写死。 |
