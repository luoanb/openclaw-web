## 1. 背景与问题

`demos/webcontainer-openclaw` 当前已完成 **WebContainer 内安装 `openclaw` 并执行 CLI 探测** 的 PoC，输出为只读日志流、单按钮流程。若要在浏览器内更贴近「本地开发体验」，需要：

- **可管理的终端式交互**：不仅执行固定脚本，还要能发起、观察、（在能力范围内）控制容器内命令，以便排障、尝试子命令、运行简单 dev 任务。
- **可预览容器内 HTTP 服务**：当用户在容器内启动 Web 服务（如静态站、Vite dev、OpenClaw 相关 HTTP 能力若可用）时，宿主页面需能 **展示该服务页面**，而不是仅看 stdout。

不做则 demo 停留在「一键跑通 + 打日志」，无法支撑「在沙箱里试跑一个带 UI 的进程」类场景，与产品化体验差距大。

## 2. 目标与成功指标

| 目标 | 说明 | 建议指标（可再定） |
|------|------|-------------------|
| 终端可管理 | 用户可在 PoC 页面内对容器内命令进行「发起 / 查看输出 / 基本生命周期管理」 | 核心路径可用率（本地手测清单全绿）；关键错误有可读提示 |
| 服务可预览 | 容器内监听端口的服务可被宿主以 iframe 或等价方式展示 | 常见 dev server 场景下能打开预览 URL 并可见页面加载 |

**北极星（可选）**：在支持跨源隔离的浏览器中，用户无需离开 demo 页即可完成「执行命令 → 看到终端输出 →（若适用）打开容器内 Web 页」闭环。

## 3. 范围

### 3.1 本期包含（In Scope）

- **终端指令管理（Must）**
  - 在宿主 UI 提供 **命令输入与执行**（至少：可输入 shell 命令行 → 在 WebContainer 内 `spawn` 执行；具体解析策略见开放问题）。
  - **输出展示**：stdout/stderr 流式追加到终端视图（可滚动、可清空或等价）。
  - **基本管理**：区分「当前/最近一次进程」与历史；**进程退出码**可见；在 SDK 能力允许下支持 **中止运行中进程**（如 `process.kill`）。
  - **可选增强（Should）**：最近若干条命令历史、复制日志、简单「预设命令」快捷按钮（如 `npx openclaw --help`）。

- **容器内 Web 服务展示（Must）**
  - 当容器内进程在约定端口提供 HTTP 服务时，宿主能够 **获得可访问的预览 URL** 并 **内嵌展示**（iframe 或 WebContainers 官方 `preview` 能力，技术选型在实现 Spec 中确定）。
  - 用户能 **看到服务已就绪的提示**（含端口/链接），并在页面内打开预览区域（非仅外链新标签，除非安全策略强制外链）。

### 3.2 本期不包含（Out of Scope）

- 完整 OpenClaw **网关、全通道、真实 LLM 代理**（与 [可行性研究](../research/feasibility-openclaw-webcontainers.md) 及现有 PoC Spec 一致）。
- 修改 `demos/openclaw` 源码；生产级多租户、登录鉴权、审计。
- 在 WebContainer 内复刻整仓 monorepo workspace（除非单独决策）。
- **真 TTY/PTY 全特性**（若平台仅支持 `spawn` 管道，则本期以管道 + 简单行为模拟终端，不承诺与本地终端 100% 一致）。

## 4. 用户与场景

### 4.1 角色概览

| 角色 | 目标 |
|------|------|
| 内部研发 / 技术产品 | 验证 OpenClaw 在 WC 内的行为、向他人演示 |
| 社区或贡献者 | 在浏览器沙箱中尝试命令，减少本机环境差异 |

### 4.2 用户故事

| 角色 | 我想要 | 以便于 |
|------|--------|--------|
| 研发 | 在页面里输入并执行容器内命令，并看到完整输出 | 排障和尝试不同 CLI 参数 |
| 研发 | 在容器里启动一个 web 服务后，在同一页看到页面效果 | 验证 HTTP 层行为而不仅看日志 |
| 研发 | 停止长时间运行的命令或重新执行 | 避免卡死或重复刷新整页 |

### 4.3 核心用户旅程

1. 用户打开 demo（`crossOriginIsolated` 就绪）→ WebContainer 已 boot（可与现有一致：首次操作再 boot）。
2. 用户在 **终端区** 输入命令（如 `npm run dev` 或项目内脚本）→ 输出流式显示 → 可中止（若支持）。
3. 若命令启动了监听端口的服务 → 系统或用户触发 **预览** → **预览区** 加载容器内 URL（或官方 preview URL），用户可见页面。
4. 异常：命令失败、端口冲突、预览跨域 → 界面展示明确错误与指向可行性/PoC 文档的说明。

## 5. 功能需求

### 5.1 终端指令管理

- **描述**：提供面向 WebContainer 的命令执行与输出管理能力，替代「仅单按钮固定流水线」；保留 PoC 一键安装探测可作为快捷入口（可选保留按钮）。
- **优先级**：Must
- **验收标准**：
  - [ ] 用户可输入至少一行命令并执行，stdout/stderr 在界面中文本展示且随输出自动滚动（或可手动滚动到底）。
  - [ ] 执行结束显示退出码；执行中可对运行中进程执行中止（若当前 `@webcontainer/api` 对应该进程可用）。
  - [ ] 并发行为有定义：默认策略为「一次一条交互式主会话」或「允许多个 spawn 但 UI 分标签」——**需在实现前在开放问题中收口并写入界面说明**。
- **边界与异常**：命令为空、boot 失败、`crossOriginIsolated` 为 false 时提示与现有一致或更明确。
- **建议 Epic/Story 拆分提示**：终端 UI + spawn 封装一层；历史/中止为独立 story。

### 5.2 容器内 Web 服务预览

- **描述**：识别或配置容器内 HTTP 服务地址，并在宿主页面内嵌展示。
- **优先级**：Must
- **验收标准**：
  - [ ] 在典型场景（如文档教程中的 `server-ready` 或等价事件）下，服务启动后用户可得 **可点的预览入口**，并在页面内 **iframe（或 API 规定的 preview 组件）** 看到页面内容。
  - [ ] 若服务未启动或 URL 无效，预览区显示占位或错误状态，而非白屏无提示。
- **边界与异常**：混合内容、COOP/CEP 与 WebContainer 文档中关于 preview 的限制；若必须新窗口打开，需在 PRD 修订中说明并更新验收。
- **建议 Epic/Story 拆分提示**：端口/URL 发现与「预览壳」UI 分 story。

### 5.3 PoC 回归与兼容

- **描述**：不破坏现有「安装 openclaw + `--help`」能力；可作为预设命令或首屏引导保留。
- **优先级**：Should
- **验收标准**：
  - [ ] 仓库根 `pnpm --filter webcontainer-openclaw` 的 build/tsc 仍通过 CI 或本地脚本约定。
- **边界与异常**：无。

## 6. 数据与状态（若适用）

- **会话状态**：WebContainer 单例策略保持与现有一致（每页一次 `boot`）。
- **终端状态**：当前进程引用、输出缓冲区（注意内存上限，大日志可截断或仅保留尾部——开放问题）。
- **预览状态**：当前预览 URL、加载状态（loading / error / ready）。

## 7. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 输出日志建议限制单行长度或总长度，避免巨量输出卡死主线程（具体阈值实现阶段定）。 |
| 安全 | 运行于用户浏览器沙箱；不向宿主持久化敏感令牌；预览 URL 仅来自 WC 提供的事件或用户明确配置的端口。 |
| 兼容 | 与 Vite COOP/COEP 配置兼容；目标浏览器为官方支持的 Chromium 系为主。 |
| 可访问性 | 终端区与预览区有语义或可聚焦区域；关键操作用 `aria-live` 或等价提示（延续现有 `aria-live` 思路）。 |
